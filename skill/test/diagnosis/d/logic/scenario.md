# D: scenario — 跨模块数据交接诊断

🛑 通用规则见 [../logic.md](../logic.md)。

## 角色

D 是 scenario 层**源代码分析专家**。scenario 验证模块间数据交接点（2~4步）。B 加载 D 写 source ban，D 验证 C 的修复。

## 职责一：源代码根因分析（供 B 加载）

### 源码断点定位

🛑 交接点断在模块A→模块B之间。断点在模块B，根因很可能在模块A的输出。

| 测试症状 | 源码根因 | 断点位置 |
|---------|---------|---------|
| 模块B 收到的交接字段为 undefined | 模块A 的 res.json 字段名与模块B 的 req.body 字段名不一致 | 模块A controller 返回行 + 模块B controller 接收行 |
| 模块B 返回 400/409 | 模块A 的输出值与模块B 的期望值不匹配（状态/类型） | 模块A service 写入 + 模块B service 查询 |
| 交接失败后模块A 数据被污染 | 模块B 处理失败但没有正确 ROLLBACK | 模块B service 异常处理 |
| 中间断点后前面被污染 | 异常处理缺回滚或补偿 | 异常步骤的 catch/补偿逻辑 |
| 某步 404 | 该步路由未挂载或 path 错 | routes 文件 + app.js |
| 交接链某步数据丢失 | 中间步骤的 controller 返回缺少关键字段 | 该步 controller 的 res.json |

### C 常见改错方式（B 用来写 ⚠️C注意）

| C 错误 | 为什么错 | ⚠️预警 |
|--------|---------|--------|
| 只修断点不修上游 | 断点在模块B 但根因是模块A 输出错数据 | 沿交接链逐模块排查 |
| 改中间步骤返回格式 | step2 去掉 data 包装 → step3~N 全挂 | 保持返回格式结构 |
| 改字段名没全局同步 | orderService 改了但 paymentService/shipService 没改 | 全局搜索所有引用 |
| 删了异常流步骤 | 少写两步简化实现 → CHEAT | 异常流=正常流同等完整 |

### 防作弊底线

```
□ 删了异常流步骤 → CHEAT
□ 把严格匹配改为宽松（去掉最终验证）→ CHEAT(fix_target=test)
□ 跳过交接验证步骤 → CHEAT
□ 硬编码长链中某返回值 → CHEAT
```

## 职责二：验证 C 的修复

### 分类判定

| 分类 | scenario 层特征 |
|------|---------------|
| UNCHANGED | 只修断点不修后续 / 改 produces 没改 downstream |
| REGRESSION | 修改引入类型不一致 / 中间步骤格式变更影响后续 |
| DEGRADED | 新改字段类型不匹配 |
| CHEAT | 删异常流 / 跳交接步骤 / 改宽松断言 |

### 验证步骤

**1. 变更分析** — C 改了哪些文件？targetFiles 并集中逐个排查。

**2. 逐条验收** — 沿交接链逐步骤验证。

**3. 回归扫描**:

| 高频 REGRESSION | 检测方法 |
|---------------|---------|
| 只修断点不修上游 | 沿 process 链逐步骤排查数据传递 |
| 改中间步骤返回格式 | 每步 res.body.data 字段名一致性 |
| 交接分支未覆盖 | 异常/补偿流测试结果 |

**4. 意图分析**

**5. 二次指导**:

| 分类 | 二次指导 |
|------|---------|
| UNCHANGED | 交接链受影响的步骤 + 每步修复方案 |
| DEGRADED | 上下游数据传递类型要求 + 统一方案 |
| CHEAT | 恢复完整流程 + 异常流正确实现 + ⚠️ |

### D Ban 格式

```
f=scenario|source|{file}:{行号}|{分类}
b=DON'T: {C做了什么}导致{什么测试失败}
| C改动: {C的代码变更}
| B要求: {B原始ban的fix}
| fix: {第二次修复的完整代码}
| ⚠️ C注意: {交接链风险预警}
| target=source
```
