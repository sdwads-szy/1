# C repair: 质量层

D 判定分类后 C 的二次修复策略。适用于 nfr。

## UNCHANGED — 同 ban 再命中

| C 常见失误 | 纠正 |
|----------|------|
| 加参数化查询漏了一处 | 逐条 SQL 全局搜索字符串拼接 |
| SQL 注入加固只改了一处 | 全局搜索 `"SELECT.*\+ req\.` 找所有拼接点 |
| XSS 漏了输出点 | 全局搜索 `res.send` / `res.json` 确认所有输出 |
| CORS 白名单配错 | 确认 origin 列表完整 |
| Token 检查漏了一处 | 全局搜索 `jwt.verify` 所有调用点 |

## REGRESSION — 原来 PASS 现在 FAIL

| C 常见失误 | 纠正 |
|----------|------|
| 改中间件导致正常请求失败 | 检查中间件条件限定范围（如限流只在特定路由） |
| 加 body size limit 影响正常上传 | 检查 limit 值是否合理 |
| 改 CORS 导致前端无法请求 | 确认 origin 与前端地址一致 |

## DEGRADED — 新错误

| C 常见失误 | 纠正 |
|----------|------|
| 新加缓存 TTL 过短 | 调整 TTL 为合理值 |
| 参数化后 LIKE 查询失效 | `%keyword%` 放在参数值中，非 SQL 中 |
| XSS 转义函数用错 | 用成熟的转义库（如 `escape-html`） |

## CHEAT

| 常见作弊 | 纠正 |
|---------|------|
| 降低 QPS 阈值 | 恢复原阈值 |
| 把攻击载荷改为正常输入 | 恢复测试（不能改测试文件） |
| 删限流中间件 | 恢复 |
| 加 try-catch 吞掉注入异常 | 恢复正确拦截 |
