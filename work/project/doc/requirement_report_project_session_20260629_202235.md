## 三、核心业务流程

### 3.1 买家主链路（C端核心闭环）

```
注册/登录 → 浏览类目/搜索商品 → 查看商品详情 → 选择SKU加入购物车
    → 进入购物车(管理商品/凑单) → 结算页(选地址/选优惠券/确认金额)
    → 提交订单 → 选择支付方式(微信/支付宝) → 完成支付
    → 等待发货 → 确认收货 → (如有问题)申请退款/退货
```

**关键节点说明**：
- **搜索介入点**：浏览类目无法快速定位时，吸顶搜索框全场可触达
- **购物车→结算**：支持全选/部分结算，库存不足/下架商品自动置灰拦截
- **支付超时**：30分钟未支付自动关单释放库存，订单状态变为"已取消"
- **退款起点**：已支付→已发货状态下可申请退款，走商家审核→平台退款→到账闭环

### 3.2 商家主链路（B端经营闭环）

```
注册商家账号 → 提交资质(营业执照/身份证/银行账户) → 等待平台审核
    → 审核通过开通店铺 → 编辑店铺信息 → 发布商品(选类目/填信息/设SKU/定价)
    → 等待商品审核 → 审核通过上架销售
    → 接收订单 → 发货 → (用户申请退款)审核退款申请
```

**关键节点说明**：
- **双重审核**：商家资质审核(OCR+人工) + 商品审核(机审敏感词/图片违规+人工抽检)
- **商品编辑**：增量更新，每次编辑自动生成快照留痕
- **退款审核**：低风险订单规则引擎自动通过，高风险人工复核

### 3.3 平台运营主链路

```
审核商家入驻 → 审核商品上架 → 监控订单履约 → 处理纠纷仲裁 → 查看经营报表
    → 管理用户(封禁/解封) → 管理优惠券(创建/发放) → 对账结算
```

## 四、关键决策与业务规则

### 4.1 信任与审核体系

| 环节 | 机制 | SLA |
|------|------|-----|
| 商家入驻 | 资质OCR自动识别(百度OCR, 准确率≥85%) + 人工复核 | ≤1工作日 |
| 商品上架 | AI图片违规检测(阿里云内容安全) + 敏感词机审 + 人工抽检 | ≤4小时 |
| 用户风控 | 设备指纹(FingerprintJS) + 行为分析规则引擎, 静默反欺诈 | 实时 |
| 违规惩罚 | 违规商品下架 + 商家扣分/冻结/清退三级处罚体系 | — |

### 4.2 逆向流程与 SLA

| 流程 | 状态流转 | SLA |
|------|---------|-----|
| 订单取消 | 待支付 → 已取消 (用户主动/30分钟超时自动) | 超时触发≤1分钟 |
| 退款申请 | 已支付 → 退款申请 → 商家审核 → 退款中 → 已到账 | 商家审核≤48h, 到账≤24h(T+1) |
| 退款校验 | 金额不可超原支付额, 校验原支付凭证幂等, 防重复退款 | — |
| 商品下架 | 已购买用户不受影响(订单快照), 未售SKU立即不可见 | 实时 |

### 4.3 资金与结算规则

| 维度 | 规则 |
|------|------|
| 支付渠道 | 微信支付 + 支付宝 H5/扫码, 一期不设钱包/余额 |
| 对账周期 | 日对账 + 月对账, 异常支付(掉单)人工补录 |
| 佣金模式 | 平台按交易额百分比抽佣(商家入驻时配置commission_rate) |
| 退款资金 | 原路退回, 退款金额从商家待结算款扣除 |
| 优惠券承担 | 一期平台承担(手动发放), 二期支持商家券(商家承担) |

### 4.4 订单状态机

```
待支付 ──(用户支付)──→ 已支付 ──(商家发货)──→ 已发货 ──(用户签收)──→ 已完成
   │                      │                    │
   └(超时30min/用户取消)   └(申请退款)           └(拒收/申请退货)
         ↓                      ↓                    ↓
      已取消              退款中→已退款         退款中→已退款
```

```
users (1) ──< addresses (N)
users (1) ──< carts (N) >── skus
users (1) ──< orders (N) >── order_items (N) >── skus
users (1) ──< payments (N)
users (1) ──< user_coupons (N) >── coupons

shops (1) ──< products (N) >── skus (N)

shops (1) ──< refunds (N)

categories (1) ──< categories (N) — 自引用3级树

products (1) ──< audit_logs (N)

orders (1) ──< payments (N)
orders (1) ──< refunds (N)
```

### 6.2 关键数据约束

- **UNIQUE 约束**:
  - `users.phone` — 手机号全局唯一
  - `carts(user_id, sku_id)` — 同一用户同一SKU仅一条购物车记录
  - `orders.order_no` — 订单号全局唯一
  - `payments(order_id, status='pending')` — 同一订单仅一条待支付流水
  - `refunds(order_id, status='applied')` — 同一订单仅一条进行中退款
  - `user_coupons(user_id, coupon_id)` — 同一用户同一券仅可领一次

- **外键级联**:
  - `users.id → orders.user_id` (RESTRICT — 有订单不可删用户)
  - `orders.id → payments.order_id` (CASCADE — 删订单级联删支付)
  - `orders.id → refunds.order_id` (CASCADE)
  - `products.id → skus.product_id` (CASCADE — 删商品级联删SKU)

- **状态机流转 (不可逆)**:
  - 订单: `pending → paid → shipped → completed` (正向不可逆)
  - 退款: `applied → approved → processing → completed`
  - 支付: `pending → paid` 或 `pending → closed`

- **加密与脱敏**:
  - `users.phone` AES-256 加密存储, 列表脱敏展示(138****1234)
  - `users.password` bcrypt 哈希
  - `merchant_qualifications.bank_account` AES-256 加密
  - 支付回调签名 RSA 验证, 100% 覆盖

- **快照与审计**:
  - `order_items.sku_snapshot` JSON 快照 — 下单时固化商品信息(价格/规格), 后续商品变更不影响历史订单
  - `product_snapshots` — 商品每次编辑自动生成完整快照(trigger)
  - `audit_logs` — 商品审核日志全量留存
